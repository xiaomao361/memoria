# Memoria 流水记录能力 — 种子

## 缘起

2026-06-21 D2。毛仔每天报告锻炼，姐每次说“记好了”，但连续三周的锻炼，前两周一条没落。Memoria 和 `current-status.md` 混用导致数据丢、查不到、上下文乱。

毛仔纠正姐两轮，没生气。说“发现问题，解决问题”。最初提议做一个独立的记录系统。

2026-06-21 继续讨论后，决定暂时不增加新的系统：在 Memoria 内增加独立的流水记录能力，但不把流水混入现有记忆。

## 核心问题

- Memoria 现有记忆存事实和锚点，不适合高频时序数据。每天一条锻炼会污染记忆上下文。
- `current-status.md` 依赖手动维护，容易漏、重复、不一致。
- 查询困难，需要翻多个地方才能拼出完整记录。
- 不同场景字段不同。锻炼有步数和时长，饮食有餐次和食物，不能只保存成一段文本。

## 已确定方向

- 不新建独立项目，能力放在 Memoria 内。
- 继续使用 SQLite，不引入专门的时序数据库。
- 新增一张通用流水表，和现有记忆表分开。
- 只做时序记录需要的能力：新增和查询为主，支持基础统计。
- 各场景共用固定字段，不同内容放在可查询的结构化数据中，不能保存成随意文本。
- 用 `type` 区分锻炼、饮食、睡眠等场景；每种类型有自己的字段规则和版本。
- 某类数据以后需要复杂操作时，再建立专用表并迁移该类历史数据。
- 流水不会直接参与 Memoria 的记忆搜索、重要性判断和整理。
- 流水中形成的长期趋势，可以提炼为一条真正的 Memoria 记忆。

## 为什么仍然放在 Memoria

这些数据本质上也是“需要被可靠记住的事实”，只是读取方式不同：

- 记忆主要按含义召回。
- 流水主要按用户、类型和时间查询。

因此不必再造一个系统，但必须保持两套数据边界，不能简单给现有记忆增加一个类型。

## 第一版数据形态

一张通用流水表，至少包含：

- 属于谁
- 记录类型
- 实际发生时间
- 结构化数据
- 结构版本
- 可选备注
- 记录来源
- 写入时间
- 防重复标识

结构化数据示例：

- 锻炼：项目、步数、时长、次数、距离
- 饮食：餐次、食物、分量、热量
- 睡眠：入睡时间、起床时间、时长、质量
- 身体状态：体重、血压、体温、症状
- 习惯：习惯名称、是否完成、完成量

## 第一版范围

- 新增一条流水。
- 按用户、类型和时间范围查询。
- 查询今天、最近一段时间的记录。
- 对明确的数字字段做基础汇总。
- 防止相同记录被重复写入。
- 首个完整模板只做锻炼；其他类型用于验证通用性，不同时做复杂分析。

第一版暂不做：

- 独立应用。
- 真正的时序数据库。
- 为每种类型建立专用表。
- 多用户产品化界面。
- 复杂修改流程和高级分析。

## 版本判断

建议作为 Memoria `v6.11.0`，不定为 `v7.0`。

这是新增一类独立能力，但没有推翻现有记忆模型、存储方式或使用方式。只有未来 Memoria 从“记忆系统”正式扩大为完整的个人数据层，并统一重做入口和定位时，才适合升为 `v7.0`。
